## 1. 后端目标

后端第一阶段负责支撑以下能力：

- 获取小程序用户身份。
- 保存和读取用户画像。
- 管理项目机会卡。
- 生成项目推荐。
- 保存收藏和历史记录。
- 生成 PRD、MVP 清单、项目对比报告。
- 管理额度、会员和后续支付。
- 承接市场雷达数据整理。

MVP 阶段优先使用 CloudBase 云函数和云数据库，不单独引入传统服务器。

## 2. 目录规划

建议项目后端目录如下：

```text
cloudfunctions/
  _template/
    package.json
  getUserContext/
  saveUserProfile/
  recommendProject/
  generatePRD/
  generateMVPPlan/
  compareProjects/
  favoriteOpportunity/
  marketRadar/
  payment/
```

`_template` 不是正式云函数，只作为后续创建云函数时复制依赖和脚本的模板。

正式云函数应按功能拆分，每个函数目录独立维护自己的 `package.json` 和依赖。微信开发者工具或 CloudBase 部署时，通常按单个云函数目录上传。

## 3. 云函数类型选择

MVP 阶段默认使用 Event Function。

原因：

- 小程序端可以直接通过 `wx.cloud.callFunction` 调用。
- 不需要额外配置 HTTP 网关。
- 小程序用户身份会自动注入云函数上下文。
- 更适合推荐、收藏、生成、保存这类业务动作。

Event Function 基础结构：

```js
const cloud = require("wx-server-sdk");

cloud.init({
  env: cloud.DYNAMIC_CURRENT_ENV
});

exports.main = async (event, context) => {
  const wxContext = cloud.getWXContext();

  return {
    ok: true,
    openid: wxContext.OPENID
  };
};
```

暂不使用 HTTP Function，除非后续需要开放浏览器 API、SSE、Webhook 或第三方回调。

## 5. 第一批云函数建议

### getUserContext

用途：

- 验证云开发环境是否接通。
- 获取当前用户 `OPENID`。
- 后续用于调试用户身份。

输入：

- 无。

输出：

- `openid`
- `appid`
- `unionid`

### saveUserProfile

用途：

- 保存用户画像问卷。
- 支持用户后续重新推荐。

集合：

- `users`
- `user_profiles`

### recommendProject

用途：

- 根据用户画像推荐 3-5 个项目机会。
- 初期可从 `opportunities` 集合中按标签筛选。
- 后续接 AI 做推荐解释和机会卡补全。

集合：

- `user_profiles`
- `opportunities`
- `generated_assets`

### generatePRD

用途：

- 根据机会卡生成 PRD。
- 生成结果保存到 `generated_assets`。

集合：

- `opportunities`
- `generated_assets`
- `quotas`

### generateMVPPlan

用途：

- 根据机会卡生成 MVP 清单。
- 输出 7 天、14 天、30 天计划。

集合：

- `opportunities`
- `generated_assets`
- `quotas`

### compareProjects

用途：

- 对比 2-3 个机会卡。
- 输出开发难度、预算、人力、变现速度、风险和推荐结论。

集合：

- `opportunities`
- `comparisons`
- `generated_assets`

### favoriteOpportunity

用途：

- 收藏和取消收藏机会卡。
- 检查免费用户收藏上限。

集合：

- `favorites`
- `users`

### marketRadar

用途：

- 读取市场雷达动态。
- 后续支持人工精选和半自动更新。

集合：

- `market_signals`

### payment

用途：

- 后续处理会员和单次付费。
- MVP 早期可先不实现真实支付。

集合：

- `orders`
- `quotas`
- `users`

## 7. 权限原则

小程序端可以读公开内容，但敏感写入和跨集合操作放到云函数：

- 机会榜、公开机会卡：可考虑客户端只读。
- 用户画像、收藏、历史记录：建议通过云函数写入。
- 额度扣减、会员状态、订单：必须通过云函数写入。
- AI Key、支付 Key、第三方平台 Key：只放云函数环境变量，不进入小程序前端。

云函数中使用：

```js
const wxContext = cloud.getWXContext();
const openid = wxContext.OPENID;
```

不要让前端传入 `openid` 作为可信身份。

## 8. AI 接入策略

MVP 阶段建议先支持 OpenAI-compatible 接口，例如 DeepSeek、通义兼容接口或其他模型服务。

原则：

- 前端不直接调用模型。
- 模型 Key 只放云函数环境变量。
- AI 输出必须结构化。
- 生成失败不扣额度。
- 相同输入短时间内可缓存。

后续如果使用 CloudBase 内置 AI，需要先确认当前环境是否有 Token Credits 或小程序成长计划资源，再启用模型组。不要在未确认资源包状态前直接写死 CloudBase AI 模型调用。

## 11. 并行开发注意事项

你会使用 git worktree 多窗口并行开发。后端并行时需要注意：

- 不同 worktree 可以同时写不同云函数。
- 不要多人同时部署同名云函数，否则云端会互相覆盖。
- 如果必须并行调试同一能力，可临时使用函数名后缀：
  - `recommendProject_tab1`
  - `recommendProject_ai`
  - `generatePRD_draft`
- 合并到主分支后，再统一改回正式函数名。
- 数据库集合是共享环境资源，测试数据需要带 `created_by`、`source` 或 `debug_tag`，方便清理。

## 12. 推荐开发顺序

1. 创建 `getUserContext`，验证云函数和 OPENID。
2. 创建 `saveUserProfile`，打通用户画像保存。
3. 创建 `opportunities` 种子数据。
4. 创建 `recommendProject`，先做规则推荐。
5. 创建 `favoriteOpportunity`，打通收藏。
6. 创建 `generatePRD` 和 `generateMVPPlan`，接入 AI。
7. 创建 `compareProjects`。
8. 创建 `marketRadar`。
9. 后续再接会员、额度和支付。

## 13. 暂不做

MVP 后端暂不做：

- 独立 HTTP API 服务。
- 复杂爬虫系统。
- 真实微信支付。
- 多租户后台权限。
- 全自动市场数据抓取。
- CloudRun 长任务服务。
