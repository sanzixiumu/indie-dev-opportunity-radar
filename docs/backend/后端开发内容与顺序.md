# 后端开发内容与顺序

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 按前端 5 Tab、功能文档和整体产品设计，明确 CloudBase 后端第一阶段要开发的内容和推荐顺序。

**Architecture:** MVP 阶段使用微信小程序 + CloudBase Event Function + CloudBase 文档数据库。小程序端通过 `wx.cloud.callFunction` 调用云函数；公开内容可读，用户资产、额度、AI Key、支付和跨集合写入全部放在云函数中处理。

**Tech Stack:** 微信原生小程序、CloudBase 云函数、CloudBase 文档数据库、`wx-server-sdk`、DeepSeek/OpenAI-compatible AI API、`zod`、`dayjs`、`nanoid`。

## 1. 前端功能对后端的要求

当前前端信息架构是 5 个 Tab：

- `找项目`：用户画像问卷、AI 推荐、机会卡详情、收藏、对比、生成入口。
- `机会榜`：榜单列表、榜单详情、机会卡跳转、运营内容更新。
- `市场雷达`：平台动态、市场信号、AI 分析、由动态生成机会卡。
- `工作台`：项目对比、PRD、MVP 清单、开发 Prompt、验证计划和额度展示。
- `我的`：用户身份、收藏、历史生成记录、画像、会员、订单、反馈。

因此后端不能只做 AI 生成函数，还需要先建立稳定的数据闭环：

1. 用户身份和画像。
2. 机会卡、榜单、市场信号这些公开内容。
3. 收藏、对比、生成记录这些用户资产。
4. 推荐、PRD、MVP、对比报告这些 AI 能力。
5. 额度、会员、订单和支付。
6. 反馈、投稿和运营更新。

## 2. 后端边界原则

- 小程序端不可信任 `openid` 入参，所有用户身份从 `cloud.getWXContext()` 获取。
- AI Key、支付 Key、内部密钥只放 `.env`、云函数环境变量或 CloudBase 配置，不进入前端。
- 机会榜、市场雷达摘要、基础机会卡可作为公开读内容；收藏、画像、历史、额度、订单必须走云函数。
- 生成失败不扣额度；重复查看已生成内容不扣额度；相同输入短时间内优先复用缓存。
- MVP 不做复杂爬虫，不做完整运营后台，不做 HTTP API 服务；优先使用 Event Function。
- CloudBase 数据库集合需要先创建，再写入数据；不要假设 `db.collection(...).add()` 会自动创建集合。

## 3. 数据集合开发顺序

### 第一批：支撑核心闭环

- `users`：用户基础信息、会员状态、最近登录时间。
- `user_profiles`：画像问卷答案和画像摘要。
- `opportunities`：机会卡主表。
- `favorites`：用户收藏。
- `generated_assets`：PRD、MVP、对比、深度报告等生成结果。
- `quotas`：额度消耗流水和每日/月度统计依据。

### 第二批：支撑内容入口

- `rankings`：机会榜配置和榜单项目。
- `market_signals`：市场雷达动态和 AI 分析。
- `comparisons`：对比项目选择和对比报告。

### 第三批：支撑商业化和运营

- `orders`：单次付费和会员订单。
- `feedbacks`：反馈、投稿、不相关推荐反馈。
- `operation_logs`：运营录入、种子数据导入、调试数据标记。

## 4. 云函数开发顺序

### Phase 1：身份、数据库和公开内容打底

目标：让小程序可以确认云开发连通，5 个 Tab 能拿到真实数据源。

函数：

- `getUserContext`
  - 输入：无。
  - 输出：`openid`、`appid`、`unionid`、`user`、`membership`、`quota_summary`。
  - 行为：按 `openid` 查找或创建 `users` 记录。

- `listOpportunities`
  - 输入：`page`、`page_size`、`tags`、`sort`。
  - 输出：机会卡列表。
  - 行为：只返回公开字段和当前用户收藏状态。

- `getOpportunityDetail`
  - 输入：`opportunity_id`。
  - 输出：完整机会卡详情、收藏状态、是否已加入对比。

- `listRankings`
  - 输入：`category`、`page`、`page_size`。
  - 输出：机会榜卡片列表、更新时间、可见范围。

- `listMarketSignals`
  - 输入：`platform`、`page`、`page_size`。
  - 输出：市场动态列表、来源、更新时间、信息完整性标记。

验收：

- [ ] 微信开发者工具中可调用 `getUserContext` 并返回当前用户 `openid`。
- [ ] `找项目`、`机会榜`、`市场雷达` 可从云数据库读取列表数据。
- [ ] 未登录浏览公开内容不需要前端传 `openid`。
- [ ] 数据不足时返回 `limited_info: true` 或明确的空状态信息。

### Phase 2：用户画像和规则推荐

目标：先不用重 AI，也能完成“填写画像 -> 推荐 3 个机会卡”的核心链路。

函数：

- `saveUserProfile`
  - 输入：技能栈、团队规模、预算、时间、平台偏好、市场偏好、变现偏好、风险偏好。
  - 输出：保存后的画像、画像版本号。
  - 行为：校验必填字段，按 `openid` upsert `user_profiles`。

- `getUserProfile`
  - 输入：无。
  - 输出：最近一次画像和画像摘要。

- `recommendProject`
  - 输入：`profile_id` 或完整画像、`mode`。
  - 输出：3-5 个机会卡、适配评分、推荐理由。
  - 行为：MVP 先做标签匹配和多样性控制，再接 AI 补充推荐解释。

验收：

- [ ] 用户可以保存画像并重复读取。
- [ ] 极端条件下至少返回 1 个推荐，并提示可放宽条件。
- [ ] 正常画像返回 3-5 个机会卡。
- [ ] 推荐结果包含推荐理由、MVP 范围、预算估算和风险提示。

### Phase 3：收藏、对比选择和我的资产

目标：把用户在各 Tab 的操作沉淀到“我的”和“工作台”。

函数：

- `favoriteOpportunity`
  - 输入：`opportunity_id`、`action`，其中 `action` 为 `add` 或 `remove`。
  - 输出：`favorited`、`favorite_count`、`favorite_limit`。
  - 行为：免费用户最多收藏 10 个；Pro/Deep 不限。

- `updateComparisonSelection`
  - 输入：`opportunity_id`、`action`，其中 `action` 为 `add` 或 `remove`。
  - 输出：当前对比项目列表。
  - 行为：同一用户最多选择 3 个机会卡，少于 2 个时只保存不生成报告。

- `listUserAssets`
  - 输入：`asset_type`、`page`、`page_size`。
  - 输出：收藏、对比、生成记录、订单摘要、额度摘要。

- `renameGeneratedAsset`
  - 输入：`asset_id`、`title`。
  - 输出：更新后的生成记录。

- `deleteGeneratedAsset`
  - 输入：`asset_id`。
  - 输出：删除状态。

验收：

- [ ] 免费用户第 11 个收藏会被后端拒绝并返回升级提示码。
- [ ] 对比项目超过 3 个会被后端拒绝。
- [ ] 工作台能读取当前对比项目。
- [ ] 我的 Tab 能读取收藏和历史生成记录。

### Phase 4：AI 生成和额度控制

目标：支撑工作台付费转化入口，并保证生成结果结构稳定。

函数：

- `generatePRD`
  - 输入：`opportunity_id`、`user_notes`。
  - 输出：`asset_id`、结构化 PRD 内容。

- `generateMVPPlan`
  - 输入：`opportunity_id`、`user_notes`。
  - 输出：`asset_id`、7 天/14 天/30 天 MVP 计划。

- `compareProjects`
  - 输入：2-3 个 `opportunity_id`。
  - 输出：`comparison_id`、对比表、AI 总结、推荐优先级。

- `generateDeepReport`
  - 输入：`opportunity_id`。
  - 输出：`asset_id`、深度项目报告。

- `generateDevPrompt`
  - 输入：`opportunity_id`、`target_tool`，其中 `target_tool` 为 `codex`、`claude_code`、`cursor` 或 `v0`。
  - 输出：`asset_id`、开发 Prompt。

- `generateLaunchPlan`
  - 输入：`opportunity_id`。
  - 输出：`asset_id`、验证计划和冷启动动作。

共用后端能力：

- `checkQuota`：生成前检查会员、单次购买和剩余额度。
- `consumeQuota`：生成成功后扣减额度并写入 `quotas`。
- `getCachedGeneration`：相同用户、相同输入、短时间内复用生成内容。
- `callAIProvider`：统一读取 `AI_PROVIDER`、`AI_MODEL`、`DEEPSEEK_THINKING`、`DEEPSEEK_REASONING_EFFORT`。

验收：

- [ ] 生成失败不写入扣费流水。
- [ ] 生成成功自动保存到 `generated_assets`。
- [ ] 历史内容重复查看不扣额度。
- [ ] AI 输出包含固定结构，无法解析时返回可重试错误。
- [ ] PRD、MVP、对比报告都能从工作台进入，并在我的 Tab 中回看。

### Phase 5：机会榜、市场雷达和运营数据

目标：把内容获客入口稳定下来，为推荐和工作台提供市场证据。

函数：

- `getRankingDetail`
  - 输入：`ranking_id`。
  - 输出：榜单说明、筛选标准、完整项目列表、更新时间。

- `getMarketSignalDetail`
  - 输入：`signal_id`。
  - 输出：市场动态详情、AI 分析、关联机会卡。

- `createOpportunityFromSignal`
  - 输入：`signal_id`、用户补充说明。
  - 输出：机会卡草稿或生成记录。
  - 行为：MVP 可以先仅面向 Deep 或运营调试使用。

- `submitFeedback`
  - 输入：`feedback_type`、`content`、`related_id`。
  - 输出：`feedback_id`。

运营侧先不做完整后台，第一版可以通过种子 JSON、CloudBase 控制台或轻量脚本维护：

- 机会卡种子数据。
- 每周机会榜。
- 市场雷达精选动态。
- 推荐标签和可见范围。

验收：

- [ ] 机会榜至少能按分类返回数据。
- [ ] 市场动态包含来源、观察信号、AI 分析和更新时间。
- [ ] 用户可以提交反馈或投稿。
- [ ] 运营数据带 `source`、`status`、`updated_at`，方便后续清理和更新。

### Phase 6：会员、订单和支付

目标：在核心生成链路跑通后接入真实商业化。

函数：

- `getMembershipCenter`
  - 输入：无。
  - 输出：当前套餐、到期时间、剩余额度、可购买商品。

- `createOrder`
  - 输入：`product_type`、`product_id`。
  - 输出：订单号、支付参数。

- `paymentNotify`
  - 输入：微信支付回调数据。
  - 输出：处理结果。
  - 行为：校验签名，更新 `orders`、`users` 和额度。

- `grantEntitlement`
  - 输入：`order_id`。
  - 输出：权益生效结果。
  - 行为：只允许支付成功订单触发。

验收：

- [ ] 免费版、年费会员、单次付费的权益判断以后端为准。
- [ ] 支付成功后权益立即生效。
- [ ] 订单重复回调不会重复发放权益。
- [ ] 产品侧不保存银行卡等敏感支付信息。

## 6. 每个云函数的统一返回格式

成功：

```json
{
  "ok": true,
  "data": {},
  "request_id": "req_xxx"
}
```

失败：

```json
{
  "ok": false,
  "error": {
    "code": "QUOTA_NOT_ENOUGH",
    "message": "当前额度不足",
    "action": "upgrade"
  },
  "request_id": "req_xxx"
}
```

常用错误码：

- `UNAUTHENTICATED`：需要登录或云函数未获取到 `OPENID`。
- `VALIDATION_ERROR`：入参不完整或类型错误。
- `NOT_FOUND`：机会卡、榜单、生成记录不存在。
- `FORBIDDEN`：访问了不属于自己的用户资产。
- `FAVORITE_LIMIT_REACHED`：免费用户收藏超限。
- `COMPARISON_LIMIT_REACHED`：对比项目超过 3 个。
- `QUOTA_NOT_ENOUGH`：额度不足。
- `AI_GENERATION_FAILED`：AI 生成失败，允许重试且不扣额度。
- `PAYMENT_REQUIRED`：需要单次付费或会员权益。

## 8. 优先级结论

最优先开发顺序是：

1. `getUserContext`，先验证云开发、用户身份和用户表。
2. `opportunities` 种子数据、`listOpportunities`、`getOpportunityDetail`，先让找项目有真实内容。
3. `saveUserProfile`、`recommendProject`，打通核心推荐闭环。
4. `favoriteOpportunity`、`updateComparisonSelection`、`listUserAssets`，让用户动作沉淀到我的和工作台。
5. `generatePRD`、`generateMVPPlan`、额度控制，打通第一批 AI 生成和转化入口。
6. `rankings`、`market_signals` 相关函数，补齐机会榜和市场雷达内容入口。
7. `compareProjects`、`submitFeedback`，完善决策和反馈闭环。
8. `orders`、`payment`、会员权益，最后接真实支付，避免在核心链路未验证前过早投入支付复杂度。
