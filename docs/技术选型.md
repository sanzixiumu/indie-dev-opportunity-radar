# 技术选型文档

产品名称：独立开发者项目机会雷达  
产品形态：微信小程序  
文档版本：v0.1  
日期：2026-06-01

## 1. 选型目标

本产品第一阶段目标是快速验证“独立开发者项目推荐 + 机会榜 + 市场雷达 + PRD/MVP 生成 + 付费转化”是否成立。因此技术选型优先考虑：

- 小程序上线速度。
- 微信生态能力集成成本。
- AI 生成能力接入便利性。
- 会员、支付、额度和历史记录的稳定性。
- 后续扩展市场雷达、运营后台和多端能力的空间。

## 2. 总体推荐方案

推荐采用：

- 小程序端：微信原生小程序 + TypeScript + TDesign Miniprogram。
- 后端：腾讯云 CloudBase 云开发 + Node.js 云函数。
- 数据库：CloudBase 云数据库。
- 文件存储：CloudBase 云存储。
- AI 服务：后端统一封装大模型接口。
- 支付：微信支付。
- 运营后台：MVP 阶段使用轻量后台或低代码后台，后期再建设独立 Web 后台。

该方案适合 MVP 阶段快速上线，能较好覆盖微信登录、云函数、数据库、存储、支付、定时任务和小程序端能力。

## 3. 前端技术选型

### 推荐方案

- 微信原生小程序。
- TypeScript。
- TDesign Miniprogram。
- 原生 tabBar。
- 原生分包。
- 微信开发者工具。

### 推荐理由

- 产品第一阶段主要验证业务闭环，不需要优先追求多端。
- 原生小程序对微信登录、支付、分享、订阅消息、云开发调用支持最直接。
- TDesign Miniprogram 组件成熟，能减少基础 UI 组件开发成本。
- 原生 tabBar 适合当前 5 Tab 结构：找项目、机会榜、市场雷达、工作台、我的。
- TypeScript 有利于维护用户画像、机会卡、生成记录、会员权益等结构化数据。

### 暂不推荐 Taro / uni-app 的原因

MVP 阶段不建议优先使用 Taro 或 uni-app，除非一开始就确定要同时做微信、抖音、支付宝、H5 等多端。当前产品的核心风险是需求和付费验证，不是多端覆盖。跨端框架会带来额外适配成本，可能拖慢第一版上线。

### 后续扩展判断

- 如果微信小程序验证成功，再考虑 Web/H5 版承接 SEO 和内容增长。
- 如果明确要进入抖音小程序、支付宝小程序，再评估 Taro 或 uni-app 重构/复用的收益。

## 4. 后端技术选型

### 推荐方案

- CloudBase 云函数：处理业务逻辑和 AI 调用。
- CloudBase 云数据库：保存用户、机会卡、收藏、生成记录、榜单、订单等数据。
- CloudBase 云存储：保存导出的 Markdown、PDF 或报告文件。
- CloudBase 定时触发器：更新机会榜、市场雷达和缓存内容。
- 云托管：后期用于更重的任务，如批量市场数据整理、长任务生成、运营后台服务。

### 后端服务模块

- 用户服务：微信登录、用户资料、会员状态。
- 画像服务：用户画像保存、更新和读取。
- 项目机会服务：机会卡、标签、项目库管理。
- 推荐服务：根据画像和项目库生成推荐。
- AI 生成服务：生成 PRD、MVP 清单、对比报告、开发 Prompt。
- 榜单服务：机会榜列表、详情、排序和更新。
- 市场雷达服务：平台动态、竞品观察、AI 解读。
- 收藏与历史服务：收藏、取消收藏、生成记录保存。
- 会员与支付服务：订单、权益、额度、支付回调。
- 反馈与投稿服务：用户反馈、项目投稿和运营审核。

## 6. AI 服务选型

### 接入原则

- 小程序端不直接调用大模型 API。
- 所有 AI 调用通过后端云函数统一封装。
- 模型供应商应可替换，避免业务代码与单一模型强绑定。
- AI 输出必须结构化，便于前端展示和后续评估。
- 付费功能、深度报告和长文本生成需要做额度控制。

### AI 服务模块

- recommendProject：根据用户画像推荐项目。
- generateOpportunityCard：补全机会卡。
- generatePRD：生成 PRD。
- generateMVPPlan：生成 MVP 清单。
- compareProjects：生成项目对比。
- analyzeMarketSignal：分析市场动态。
- generateDevPrompt：生成开发工具提示词。
- generateLaunchPlan：生成冷启动验证计划。

### 模型供应商建议

MVP 阶段可以优先接入 1-2 个模型供应商，后续再做多模型路由。

可选方向：

- 国内模型：DeepSeek、通义千问、腾讯混元、智谱。
- 国际模型：OpenAI、Claude，视网络、成本、合规和稳定性决定。

### 生成质量控制

- 使用固定 JSON Schema 或结构化 Markdown 模板。
- 每类生成内容有固定字段。
- 对市场数据不足的内容标注“信息有限”。
- 区分事实、推断和建议。
- 生成失败不扣减额度。
- 相同输入短时间内优先复用缓存。

## 7. 市场雷达技术方案

### MVP 阶段

不建议第一版做复杂爬虫和全平台实时数据抓取。推荐：

- 人工精选平台动态。
- 半自动脚本辅助整理公开信息。
- 运营后台录入市场动态。
- 定时任务更新榜单和精选内容。
- 每条动态保留来源、更新时间和可信度标记。

### 后续阶段

根据用户需求和商业化情况逐步接入：

- App Store 公开接口或 RSS。
- 第三方应用数据 API。
- 合规小程序数据服务。
- Product Hunt、Chrome Web Store 等公开数据源。
- 商业数据合作。

### 合规要求

- 不绕过平台反爬机制。
- 不违规抓取需要授权的数据。
- 不把未经验证的信息写成确定事实。
- 展示数据来源和更新时间。

## 8. 支付与会员技术方案

### 推荐方案

- 微信支付。
- 云函数处理下单和支付回调。
- orders 集合记录订单。
- users 或 memberships 字段记录会员状态。
- quotas 集合记录额度消耗。

### 权益控制

权益判断必须以后端为准，前端只负责展示。

需要控制的权益包括：

- 每日推荐次数。
- PRD 生成次数。
- MVP 清单生成次数。
- 对比报告次数。
- 深度报告次数。
- 收藏数量。
- 市场雷达深度分析权限。
- 导出权限。

### 扣费与扣额度规则

- 生成成功后扣减额度。
- 生成失败不扣额度。
- 重复查看历史内容不扣额度。
- 单次付费内容永久保留在“我的”中。

## 9. 运营后台选型

### MVP 阶段

推荐先使用轻量后台或低代码后台，避免投入过多前台以外的开发成本。

后台至少需要支持：

- 管理机会卡。
- 管理机会榜。
- 管理市场动态。
- 查看用户反馈。
- 查看订单和额度。
- 配置会员权益。

### 后续阶段

当内容运营规模变大后，再建设独立 Web 后台。

后期后台可采用：

- Next.js。
- React。
- Tailwind CSS 或组件库。
- 独立管理端鉴权。
- 与 CloudBase 或后端 API 对接。

## 11. 分阶段落地建议

### MVP

- 微信原生小程序。
- 5 个 Tab 基础页面。
- 微信登录。
- 用户画像问卷。
- AI 推荐 3 个项目机会。
- 机会卡详情。
- 收藏。
- 简单 PRD/MVP 生成。
- 机会榜。
- 市场雷达人工精选。
- 单次付费和基础会员。

### v1.1

- 完整项目对比。
- PRD/MVP 导出。
- 开发 Prompt 生成。
- 用户反馈闭环。
- 运营后台完善。
- 市场雷达固定更新机制。

### v2.0

- 接入合规第三方数据 API。
- 建立项目适配评分模型。
- 建立小团队可做指数。
- 建立冷启动难度指数。
- 支持项目持续跟踪。
- 评估 Web/H5 或多端扩展。

## 12. 最终结论

第一版推荐采用“微信原生小程序 + TypeScript + TDesign Miniprogram + CloudBase + Node.js 云函数”的组合。

这套方案能最快支撑核心闭环：用户填写画像、AI 推荐项目、查看机会卡、收藏、生成 PRD/MVP、付费、保存历史记录。等产品验证付费和留存后，再逐步加强市场雷达数据能力、运营后台和多端扩展。
